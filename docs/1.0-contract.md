# cargo-fresh 1.0 Contract

This page is the pre-1.0 contract checklist for cargo-fresh. The feedback
window closes on **2026-06-30**; after that the project will cut either
`1.0.0-rc.1` or go straight to `1.0.0`, depending on whether breaking-class
feedback needs an RC bake.

## Stable Surfaces

These surfaces become semver-protected at 1.0. Changing them after 1.0 requires
a major version bump unless the change is explicitly additive.

| Surface | 1.0 value | Stability rule |
|---------|-----------|----------------|
| JSON output | `schema_version = 2`, `format = "cargo-fresh-v1"` | Additive fields are allowed in 1.x; renames, removals, and type changes require 2.0 |
| Exit codes | `0` ok, `1` updates available but not applied, `2` at least one update failed, `130` SIGINT | Codes are not reused or removed within 1.x |
| CLI flag inventory | The `--help` flag set plus `completion` and `man` subcommands | Deprecations get at least one minor warning cycle before removal |
| Source-aware install behavior | crates.io, git, path, and unknown sources keep their documented update/skip strategy | Behavior-preserving fixes are allowed; semantic changes need an RC/breaking review |

## Not Stable

These surfaces intentionally stay flexible in 1.x:

- Human-readable status wording, color, alignment, and progress-row text.
- English and Chinese locale phrasing.
- `--debug` output. It is for issue reports only and may change in any release.
- Internal Rust modules and the `cargo_fresh::*` library surface. `src/lib.rs`
  exists so integration tests can share the binary's module tree.

## JSON Contract

`--format=json` writes exactly one JSON object to stdout. Status lines, errors,
prompts, progress, and `--debug` traces go to stderr.

The schema source of truth is [`docs/json-schema.json`](json-schema.json).
`schema_version = 2` is the final pre-1.0 schema break. It replaced the old
`updates_available[].binstall` field with `updates_available[].prebuilt` and
renamed the old `source_build` enum value to `source`.

Important stable scripting fields:

- `updates_available[]`: update candidates found during the check phase.
- `skipped[].reason_code`: machine-readable skip reason.
- `version_check_errors[]`: packages whose latest-version lookup failed.
- `summary.selected`, `summary.attempted`, `summary.check_errors`: policy and
  execution counts.
- `results[].install_method`: which path actually ran (`prebuilt` / `source` /
  `null`); shares vocabulary with `updates_available[].prebuilt`.
- `version`: the cargo-fresh release that produced the report (self-describing;
  not a stability discriminator — branch on `schema_version` / `format`).
- `aborted`: whether the run was interrupted.

## Feedback Wanted

The tracking issue is
[#3 "Towards 1.0 - Feedback Wanted"](https://github.com/jenkinpan/cargo-fresh/issues/3).
Useful feedback before the freeze:

- A CLI flag name or behavior that would be hard to live with in 1.x.
- A JSON field that is missing, ambiguous, or awkward to script against.
- An exit-code case that needs a distinct code.
- A real error that should have printed a better `Hint:` line.
- Mirror, proxy, private-registry, git-source, or path-source behavior that does
  not match the documented contract.

For downloader/prebuilt issues, please include:

```bash
cargo fresh --debug --check-prebuilt 2>&1 | grep debug
```

The `--debug` trace format is deliberately not stable; it is just enough context
to reconstruct downloader decisions in an issue.
